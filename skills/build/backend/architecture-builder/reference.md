# Architecture Builder — Reference

> Deep-dive material for the architecture-builder skill. Read on demand, not upfront.

## Architecture Decision Records (ADR)

Use the Nygard format for every significant architectural decision:

```markdown
# ADR-001: Use PostgreSQL as Primary Database

## Status
Accepted

## Context
We need a relational database that supports ACID transactions for financial data.
Team has strong PostgreSQL experience. Budget limits managed service options.

## Decision
We will use PostgreSQL 16 hosted on AWS RDS.

## Consequences
+ ACID transactions, mature tooling, team familiarity
+ RDS handles backups, patching, HA failover
- Vendor lock-in to AWS
- Higher cost than self-hosted at scale (revisit at 100k users)
```

**When to write an ADR:**
- Technology selection (database, message broker, framework)
- Any decision that is hard to reverse
- Any decision that affects team workflow significantly
- Decisions made under constraints that future developers won't see

**Ultra-concise alternative — Y-Statement:**
```
In the context of [use case],
facing [concern],
we decided for [option]
to achieve [quality],
accepting [downside].
```

## C4 Model — Four Levels of Architecture

| Level | Name | Shows | Audience |
|---|---|---|---|
| 1 | System Context | System + external actors + dependencies | Stakeholders |
| 2 | Container | Deployable units (app, DB, queue, cache) | Architects, devs |
| 3 | Component | Internal building blocks per container | Developers |
| 4 | Code | Classes, functions | Developers (often skipped) |

Start at Level 1 for stakeholder alignment. Drill to Level 2 for technical planning. Level 4 is typically auto-generated by IDEs.

**Rule:** Every arrow must be labeled with *what* and *why*, not just protocol.

## Scalability Deep-Dive

### Caching Strategy Layers

```
Request → CDN (L3) → Load Balancer → App (L1 in-process) → Redis (L2) → Database
```

| Layer | What | Use For | Limitation |
|---|---|---|---|
| L1 In-Process | Local memory (`lru_cache`, dict) | Config, feature flags, small lookups | Not shared across instances |
| L2 Distributed | Redis / Memcached | Sessions, API responses, aggregates, rate limits | Network latency |
| L3 CDN | Edge cache (CloudFront, Fastly) | Static assets, public API responses | Staleness |

### Caching Patterns

| Pattern | How | When | Risk |
|---|---|---|---|
| Cache-Aside | App checks cache; on miss reads DB, populates cache | Default choice | Cache miss adds latency |
| Read-Through | Cache fetches from DB on miss automatically | Cache layer manages reads | Infrastructure complexity |
| Write-Through | Write to cache and DB synchronously | Reads need fresh data | Slows writes |
| Write-Behind | Write to cache; async batch to DB | High write throughput | Data loss on cache failure |

### Database Scaling Path

1. **Connection Pooling** — Always. Use PgBouncer or SQLAlchemy pool. Naive one-connection-per-request kills databases at scale.

2. **Read Replicas** — When read:write > 5:1. Primary handles writes, replicas handle reads. Introduces eventual consistency (replication lag).

3. **Sharding** — Last resort. Single DB at vertical ceiling AND read replicas didn't solve it.
   - Range sharding: by ID/date. Simple but uneven distribution.
   - Hash sharding: even distribution but cross-shard queries impossible.
   - Directory sharding: lookup table. Flexible but lookup is bottleneck.

### Event-Driven Patterns

| Pattern | Use When | Tool |
|---|---|---|
| Pub/Sub | Notifications, cache invalidation, analytics | Redis Pub/Sub, Kafka |
| Event Sourcing | Full audit log, time-travel queries needed | Kafka, EventStoreDB |
| Saga | Distributed transactions across services | Choreography via events |
| CQRS | Read/write workloads scale differently | Separate read/write models |

**CQRS is overkill when:** CRUD app, team < 5, early-stage product.

## SOLID at Architecture Scale

| Principle | Class Level | Architecture Level |
|---|---|---|
| SRP | One class, one reason to change | One module, one business capability |
| OCP | Open for extension, closed for modification | Add features via new modules/plugins, not changing existing |
| LSP | Subtypes substitutable | API versions don't break consumers |
| ISP | No fat interfaces | Purpose-specific APIs (BFF pattern) |
| DIP | Depend on abstractions | Services communicate via events/interfaces, not direct imports |

## Security Architecture Layers

### Defense in Depth

```
1. Network     → Firewall, WAF, DDoS protection
2. Auth        → Identity verification at every entry point
3. Authz       → Least-privilege access per resource
4. Application → Input validation, output encoding, CSRF/XSS
5. Data        → Encryption at rest, column-level masking, audit logs
6. Monitoring  → Anomaly detection, alerting, incident response
```

### Zero Trust Checklist (Developer-Relevant)

- [ ] Every API endpoint requires authentication — no implicit trust from network
- [ ] Short-lived tokens (JWT expiry < 15min for sensitive ops)
- [ ] Service-to-service auth with mTLS or signed tokens
- [ ] All secrets in secret manager (not env vars in code)
- [ ] All access logged with correlation IDs
- [ ] PII classified by sensitivity; different controls per class

### Auth Architecture Decision

| Approach | When | Python Implementation |
|---|---|---|
| Session-based | Server-rendered apps, Django | Django sessions + middleware |
| JWT stateless | API-first, mobile clients | `python-jose` / `PyJWT` + FastAPI deps |
| OAuth2/OIDC | Delegated auth, SSO | `authlib` / Auth0 / Keycloak |
| API Keys | Service-to-service, simple | Header-based with rate limiting |

## Architecture Fitness Functions

Automated guardrails that prevent architectural drift.

### Python Implementation

**Import boundaries** (using `import-linter`):
```ini
# .importlinter config in pyproject.toml
[tool.importlinter]
root_packages = ["src"]

[[tool.importlinter.contracts]]
name = "Domain must not import infrastructure"
type = "forbidden"
source_modules = ["src.modules.*.domain"]
forbidden_modules = ["src.modules.*.infrastructure", "src.modules.*.api"]

[[tool.importlinter.contracts]]
name = "No cross-module imports"
type = "independence"
modules = ["src.modules.users", "src.modules.billing", "src.modules.notifications"]
```

**Complexity gates** (using `radon`):
```bash
# Fail CI if any function exceeds cyclomatic complexity of 10
radon cc src/ -n C --total-average --no-assert
```

**Dependency cycles** (using `pydeps` or `deptry`):
```bash
# Detect circular imports
deptry src/ --no-ansi
```

**Performance fitness** (using `locust` or `k6`):
```python
# locustfile.py — fail if p95 > 200ms
from locust import HttpUser, task

class APIUser(HttpUser):
    @task
    def get_users(self):
        self.client.get("/api/v1/users")
```

## The Migration Path: Strangler Fig

When evolving from monolith to services:

1. Identify a bounded context with clear ownership and minimal dependencies
2. Build the new service with its own data store
3. Route traffic through a facade (API gateway / reverse proxy)
4. Gradually migrate traffic from old to new
5. Decommission the old module when traffic is fully migrated

**Never big-bang rewrite.** Extract one service at a time.
